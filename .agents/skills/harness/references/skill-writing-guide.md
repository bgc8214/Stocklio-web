# 스킬 작성 가이드

재사용 가능한 전문 스킬은 "잘 선택되고, 읽기 쉽고, 유지비가 낮은 형태"로 써야 한다.

## 1. YAML Frontmatter로 시작한다

모든 `SKILL.md`는 YAML frontmatter로 시작한다.

- `name`: 안정적이고 저장소 친화적인 스킬 이름
- `description`: 언제 선택되어야 하는지 설명하는 한 줄 요약

예시:

```markdown
---
name: api-docs
description: 기존 코드베이스에서 엔드포인트 목록, 예시, 스키마 메모를 포함한 API 문서를 생성한다.
---

# API 문서
```

## 2. 선택 경계를 먼저 분명하게 한다

스킬 상단에는 다음이 바로 보여야 한다.

- 언제 이 스킬을 써야 하는가
- 이 스킬이 유의미하게 일하려면 어떤 입력이 필요한가

### 좋은 패턴

```markdown
## 언제 사용할까
- 기존 코드베이스에서 API 문서를 생성할 때 사용
- 엔드포인트 목록, 사용 예시, 스키마 메모가 필요할 때 사용

## 필요한 입력
- 저장소 또는 API 소스 파일
- 원하는 출력 형식
- 대상 독자와 깊이
```

## 3. 본문은 가볍게 유지한다

메인 `SKILL.md`에는 보통 다음이면 충분하다.

- frontmatter
- 목적
- 선택 경계
- 입력 계약
- 워크플로우
- 출력 계약
- 검증 메모

## 4. Why 먼저 설명한다

직관적이지 않은 규칙은 이유를 붙인다.

## 5. 출력 계약을 명시한다

출력 구조가 중요하면 파일명까지 분명히 적는다.

## 6. 긴 설명보다 예시를 쓴다

형식이나 판정 규칙은 예시로 보여주는 편이 더 명확할 때가 많다.

## 7. Progressive Disclosure

긴 설명은 `references/`로 나눈다.

## 8. 스크립트는 반복적이고 결정적인 일에만 번들링한다

- 매번 돌리는 구조 검증기
- 결정적인 리포트 정규화 도구
- 스키마 비교 보조 도구

## 9. 권장 섹션 순서

```markdown
---
name: skill-name
description: 이 스킬이 선택되어야 하는 상황을 한 줄로 요약
---

# 스킬 이름

## 언제 사용할까
## 필요한 입력
## 작업 흐름
## 출력
## 검증
## 참고 문서
```

## 10. 데이터 스키마 기준

평가에 참여하는 스킬이라면 테스트 메타데이터와 채점 결과 형식을 안정적으로 유지한다.

## 11. 무엇을 넣지 말아야 하나

- 모델이 이미 잘 아는 일반 상식
- 짧은 예시로 대체 가능한 긴 설명
- 저장소 문맥과 무관한 플랫폼 전용 가정
- 이미 다른 문서에 있는 중복 지시
