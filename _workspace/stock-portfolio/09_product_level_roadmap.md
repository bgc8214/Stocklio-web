# 제품 레벨 로드맵

현재 상태는 `MVP+`다. Numbers의 핵심 계산과 화면은 웹으로 옮겼고, 가격 자동 업데이트와 예수금/성과/백업까지 붙었다. 하지만 실제 제품이 되려면 사용자별 데이터 분리, 영구 DB, 인증, 장애 대응, import 확정 흐름, 배포 운영이 필요하다.

## 제품 레벨 정의

제품 레벨은 다음 조건을 만족해야 한다.

- 여러 사용자가 각자 자기 포트폴리오를 안전하게 관리한다.
- 데이터가 브라우저나 로컬 SQLite에만 있지 않고 운영 DB에 저장된다.
- 가격 업데이트 실패, 환율 실패, 휴장일, 수동 가격 보정이 화면과 로그에 남는다.
- Numbers/XLSX import는 미리보기, 검증, 계좌 매핑, 예수금 배분, 확정 저장 단계를 가진다.
- 총자산, 평가손익, 입출금 제외 성과, 배당/세금 반영 성과의 정의가 고정된다.
- Vercel 배포, 환경변수, 백업, 모니터링, 롤백 절차가 있다.

## Phase 1: MVP 안정화

상태: `완료`

- 화면 overflow와 입력 폼 깨짐 제거
- 모든 메뉴 모바일/데스크톱 렌더 검증
- 현재 SQLite 상태 JSON 백업/복원 안정화
- 가격 업데이트 로그와 계좌 합계 검증 유지
- README 실행/검증 명령 정리

완료 기준:

- Playwright에서 데스크톱/모바일 `bodyOverflow = false`
- `node --check app.js`, `node --check server.mjs`, migration test 통과
- 총자산과 계좌 합계 차이 0원

## Phase 2: 데이터 모델 제품화

상태: `부분 완료`

- SQLite JSON blob을 정규 테이블로 분리
- `users`, `portfolios`, `accounts`, `holdings`, `cash_balances`, `cash_flows`, `prices`, `snapshots`, `import_jobs` 설계
- 금액 계산을 UI 함수가 아니라 서버 도메인 함수로 이동
- 계산 결과에 버전과 생성 시각 저장
- 마이그레이션 스크립트에 재실행/롤백 옵션 추가

로컬 반영:

- `accounts`를 앱 상태에 추가했다.
- 기존 보유 종목과 예수금에서 계좌 레지스트리를 자동 생성한다.
- 계좌명 수정 시 보유 종목, 예수금, 현금흐름 참조를 함께 갱신한다.
- 마이그레이션 state version을 6으로 올리고 계좌 데이터를 포함한다.

남은 차이:

- 아직 SQLite 정규 테이블이 아니라 `app_state` JSON blob에 저장한다.
- 금액 계산 함수가 UI와 서버에 중복되어 있다.
- `prices`, `import_jobs`, 거래 이벤트 테이블은 아직 없다.
- 마이그레이션 재실행/롤백 UX는 아직 없다.

완료 기준:

- 같은 입력 데이터로 항상 같은 총자산/성과가 계산된다.
- UI 저장소와 서버 저장소가 분리된다.
- 테스트 데이터와 개인 데이터가 명확히 분리된다.

## Phase 3: 사용자와 보안

상태: `다음`

- Vercel 배포용 인증 도입
- 사용자별 포트폴리오 격리
- 민감 데이터 export/import 권한 제어
- 실제 계좌명, 투자자명, 보유 내역을 서버 로그에 과도하게 남기지 않도록 조정
- 백업 파일에 개인정보 포함 경고와 암호화 옵션 검토

완료 기준:

- 다른 사용자의 포트폴리오를 API로 조회할 수 없다.
- 로그인하지 않으면 개인 데이터 API를 사용할 수 없다.
- 샘플 데모와 실제 데이터 모드가 분리된다.

## Phase 4: Vercel 운영 배포

상태: `준비`

- 현재 정적 화면은 Vercel에 올릴 수 있게 `vercel.json`을 추가했다.
- 실제 제품 API는 Vercel Functions로 분리한다.
- 영구 DB는 Vercel Postgres, Neon, Supabase 중 하나로 둔다.
- Yahoo Finance 프록시와 가격 업데이트 작업은 서버리스 함수 또는 예약 작업으로 분리한다.
- 로컬 SQLite는 개발/마이그레이션 용도로만 유지한다.

권장 구조:

- `app.js`, `index.html`, `styles.css`: 정적 프론트엔드
- `api/state`: 사용자 포트폴리오 조회/저장
- `api/yahoo/chart`: 가격 프록시
- `api/automation/run`: 예약 가격 업데이트
- 운영 DB: Postgres
- 배포: Vercel

완료 기준:

- Vercel URL에서 로그인, 데이터 저장, 가격 업데이트가 동작한다.
- 배포 후 새로고침해도 사용자 데이터가 유지된다.
- 예약 업데이트 실패가 화면과 로그에 표시된다.

## Phase 5: Import 제품화

상태: `최우선 다음`

- Numbers/XLSX 업로드 UI
- import preview 화면
- 종목명/티커/시장 자동 매핑
- 미분류 예수금 계좌별 배분 확정
- 기존 데이터와 병합 또는 새 포트폴리오 생성 선택
- import job 결과 리포트 저장

완료 기준:

- 사용자가 Numbers 파일만 올려도 보유 종목, 스냅샷, 예수금 후보가 preview로 보인다.
- 저장 전 검증 오류를 사용자가 수정할 수 있다.
- 확정 전에는 운영 포트폴리오 데이터가 바뀌지 않는다.

## Phase 6: 성과 분석 고도화

상태: `부분 완료`

- 입출금 보정 수익률
- 배당, 세금, 수수료
- 계좌별/전략별/종목별 기여도
- 기간 필터: 1일, 1주, 1개월, 올해, 전체
- 원화 기준과 달러 기준 성과 분리

로컬 반영:

- 성과 기간 필터를 추가했다.
- 입금/출금 외에 배당, 세금, 수수료 현금흐름 유형을 추가했다.
- 입출금 보정 수익률 계산에서 외부 유입은 입금/출금만 반영하고 배당/세금/수수료는 투자 성과에 남기도록 분리했다.

남은 차이:

- 계좌별/전략별/종목별 기여도는 아직 상세 표로 구현하지 않았다.
- 원화 기준과 달러 기준 성과 전환은 아직 없다.
- 배당/세금/수수료가 예수금 잔액과 자동 연결되지는 않는다.

완료 기준:

- 단순 총자산 증가와 투자 성과 증가가 구분된다.
- 입출금이 큰 날에도 성과가 왜곡되지 않는다.
- Numbers보다 더 설명 가능한 성과표가 된다.

## Phase 7: 제품 경험

상태: `부분 완료`

- 종목 상세 페이지
- 계좌 상세 페이지
- 가격 업데이트 실패 해결 가이드
- 알림: 가격 업데이트 실패, 큰 변동, 스냅샷 미생성
- 사용자 지정 대시보드 레이아웃
- 모바일 사용성 개선
- 샘플 데이터 온보딩

로컬 반영:

- 계좌 관리 화면을 추가했다.
- 계좌 상세에서 보유 종목, 예수금, 최근 현금흐름을 함께 확인할 수 있다.
- 보유 종목, 예수금, 입출금은 기존 계좌 선택과 수정 모드를 지원한다.
- 대시보드 카드는 사용자가 직접 드래그로 재배치하고, 우하단 핸들로 폭/높이를 조절하며, 표시 여부를 바꾸고 기본값으로 복원할 수 있다.

남은 차이:

- 종목 상세 페이지는 아직 없다.
- 가격 업데이트 실패 해결 가이드와 알림은 아직 없다.
- 대시보드 프리셋 저장, 모바일 전용 배치, 사용자별 동기화는 아직 없다.
- 샘플 데이터 온보딩은 아직 없다.

완료 기준:

- 대시보드는 요약만 보여주고, 상세 원인은 메뉴별 페이지에서 추적 가능하다.
- 사용자가 “왜 총자산이 바뀌었는지”를 한 화면 안에서 설명할 수 있다.

## Phase 8: 운영 안정성

상태: `다음`

- API 에러 추적
- DB 백업과 복구 절차
- 가격 데이터 캐시와 재시도
- import 전후 스냅샷 비교
- 테스트 자동화: 계산, API, 브라우저 렌더, 모바일 overflow

완료 기준:

- 장애가 나도 사용자의 원본 데이터가 보존된다.
- 배포 전 검증 명령으로 핵심 회귀를 잡을 수 있다.
- 운영 데이터와 테스트 데이터가 섞이지 않는다.

## 로컬 우선 실행 순서

Vercel, 로그인, 다중 사용자 권한을 제외하면 다음 순서가 가장 효율적이다.

1. Phase 5 Import 제품화
   - 앱 안에서 XLSX 파일 선택
   - preview 생성
   - 보유/스냅샷/예수금 후보 검증
   - 기존 데이터 병합 또는 교체 선택
   - 확정 전에는 현재 포트폴리오를 변경하지 않음

2. Phase 2 계산/저장 구조 정리
   - UI/서버 중복 계산 제거
   - 공통 계산 모듈 작성
   - `accounts`, `holdings`, `cashBalances`, `cashFlows`, `snapshots` 검증 함수 분리
   - 이후 정규 테이블로 갈 수 있게 상태 경계를 정리

3. Phase 6 성과 분석 고도화
   - 계좌별/전략별/종목별 기여도
   - 원화/달러 기준 전환
   - 배당/세금/수수료 반영 방식 확정

4. Phase 7 상세 경험
   - 종목 상세
   - 계좌 상세 강화
   - 대시보드 프리셋 저장과 모바일 전용 배치
   - 가격 실패 해결 가이드
   - 삭제 되돌리기

5. Phase 8 로컬 운영 안정성
   - 계산 테스트 확장
   - 브라우저 회귀 테스트 스크립트화
   - 백업/복원 실패 케이스 검증
