# QA 전문 역할 가이드

하네스에 재사용 가능한 QA 스킬이나 QA 브리프가 필요할 때 이 가이드를 쓴다. 핵심은 "각 부분이 존재하는가"가 아니라 "서로 연결되는 경계면에서 실제로 맞는가"다.

## 1. 흔한 실패 패턴: 경계면 불일치

통합 실패는 대체로 각 부분이 따로 보면 맞아 보이는데, 연결 지점에서 서로 기대하는 모양이 다를 때 발생한다.

## 2. 통합 일관성 검증

QA는 항상 경계면 양쪽을 동시에 읽어야 한다.

### API 응답 vs 소비자 타입

1. 실제 응답 형태를 본다.
2. 소비하는 훅, 파서, 타입 정의를 본다.
3. 중첩 구조, 필드 이름, null 가능성, 배열/객체 형태를 비교한다.

### 파일 경로 vs 링크

1. 실제 라우트나 파일 구조를 정리한다.
2. 모든 링크와 네비게이션 대상을 모은다.
3. 각 대상이 실제 경로로 이어지는지 확인한다.

### 상태 전이 표 vs 런타임 업데이트

1. 허용 전이를 모은다.
2. 코드에서 실제 상태 갱신을 모은다.
3. 빠진 전이, 죽은 전이, 문서에 없는 전이를 비교한다.

## 3. QA 스킬 설계 원칙

### 양쪽을 함께 읽기

한쪽만 읽는 QA는 구조적으로 불완전하다.

### 존재 확인보다 교차 검증

"있다"보다 "맞물린다"를 확인해야 한다.

### 점진적 QA

끝에 한 번만 몰아서 보기보다, 의미 있는 모듈이 하나 완성될 때마다 일찍 확인한다.

### 실행 가능한 보고서

좋은 QA 보고서는 다음을 포함한다.

- 어떤 경계면을 봤는가
- 무엇이 어떻게 안 맞는가
- 예상 영향은 무엇인가
- 가장 작은 수정 경로는 무엇인가

## 4. QA 스킬 브리프 템플릿

```markdown
# QA 점검자

## 언제 사용할까
- 구현 후 또는 단계별 통합 시 경계면 검증이 필요할 때 사용

## 필요한 입력
- 원래 스펙 또는 요청
- producer 쪽 파일
- consumer 쪽 파일
- 승인 기준 또는 리뷰 루브릭

## 책임
- 경계면 양쪽을 비교
- 구체적인 불일치와 영향 보고
- 확인된 실패와 미확인 영역 구분

## 출력
- `_workspace/qa_report.md`
- 수리 경로가 명확하면 작은 수정 목록

## 검증
- 비교한 파일과 산출물을 명시
- blocking issue와 follow-up improvement 구분
```
