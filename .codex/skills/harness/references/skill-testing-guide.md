# 스킬 테스트와 반복 개선 가이드

Harness는 스킬 품질을 감에 맡기지 않고, 반복적으로 개선하는 엔지니어링 문제로 본다.

## 1. 테스트 모델

정성 평가와 정량 평가를 같이 쓴다.

## 2. 테스트 프롬프트 설계

테스트 프롬프트는 실제 요청처럼 들려야 한다.

- 핵심 유즈케이스 1개
- 엣지 케이스 1개
- 스킬이 필요 없는 near-miss 1개

## 3. 전문 스킬 vs 기준선

필요하면 전문 스킬 사용 버전과 더 단순한 기준선 버전을 비교한다.

## 4. Assertion 기반 평가

좋은 assertion은:

- 참/거짓이 분명하다
- 스킬의 실제 가치를 건드린다
- 두 접근법 차이를 드러낸다

## 5. 평가용 전문 역할

- `Grader`: assertion과 근거를 점검
- `Comparator`: A/B 비교
- `Analyzer`: 반복 실패와 구조적 문제 분석

## 6. 반복 개선 루프

1. 스킬 수정
2. 같은 시나리오 재실행
3. 품질, 명확성, 비용 비교
4. 원인을 일반화
5. 더 이상 의미 있는 개선이 없으면 정리

## 7. 선택 경계 검증

- should-use 요청
- should-not-use 요청

이 둘을 구분할 수 있어야 스킬 경계가 선명하다.

## 8. 실패 흐름 테스트

오케스트레이터는 최소한 하나의 실패 시나리오를 가져야 한다.

- 입력 누락
- 병렬 분기 일부 실패
- 합성 단계 상충
- 재시도 예산 초과

## 9. 검증 스크립트를 번들링할 때

- 같은 구조 체크를 매번 돌릴 때
- 출력 형식이 자동 검증할 만큼 결정적일 때
- 수작업 잡음을 줄여줄 때만 작은 검증 스크립트를 추가한다
