# 자율 실험 워크플로우

Harness는 사용자가 직접 통제하는 컴퓨터에서 반복 실험 루프를 돌리고 싶을 때 자율 실험 워크플로우를 지원할 수 있다.

## 언제 사용할까

- 작업이 단발 구현이 아니라 명시적인 반복 실험 루프일 때
- 변경 가능한 표면이 충분히 좁아 반복 수정과 리뷰가 가능할 때
- 평가 표면을 실행 중 고정해 둘 수 있을 때
- 성공 여부를 지표, 점수, 비교 규칙으로 말할 수 있을 때
- 실험이 사용자가 통제하는 실행 환경에서 돌아갈 때

## 언제 사용하지 말까

- 같은 루프 안에서 평가 표면도 계속 바뀔 때
- 변경 가능한 표면이 너무 넓어 되돌리기 기준이 흐릴 때
- 지표나 비교 규칙이 선언되지 않았을 때
- 크래시, 타임아웃, 부분 실패가 실제 운영 부작용을 낳을 수 있을 때
- 사용자가 감사할 수 없는 원격/공유 환경에서 돌릴 때

## 핵심 계약

- mutable surface: 후보가 바꿀 수 있는 파일, 파라미터, 프롬프트, 스크립트
- immutable evaluation surface: 실행 중 읽기 전용으로 유지할 코드, 데이터, 평가 기준
- baseline run: 아무 변경도 없는 기준선 실행
- comparison rule: 고정 예산, 고정 데이터셋, 고정 프롬프트 세트 같은 비교 계약
- experiment ledger: `_workspace/experiments/{run}/results.tsv`
- keep/discard policy: 후보를 채택할지, 되돌릴지, 재시도할지 규칙
- failure policy: 재시도 한도, 타임아웃 규칙, 크래시 처리

## 최소 산출물 세트

```text
_workspace/
└── experiments/
    └── {run}/
        ├── request-summary.md
        ├── baseline.md
        ├── results.tsv
        ├── candidate-{id}.md
        ├── eval-{id}.md
        └── final-summary.md
```

## 권장 루프

1. `request-summary.md`에 요청, 제약, 변경 표면, 평가 표면, 지표를 적는다.
2. 아무 변경 없는 기준선을 실행하고 `baseline.md`와 `results.tsv`에 남긴다.
3. 한 번에 하나의 제한된 후보만 제안한다.
4. 선언한 예산과 비교 규칙 안에서 후보를 실행한다.
5. 실행 직후 결과를 `results.tsv`에 기록한다.
6. 선언한 정책에 따라 채택하거나 폐기한다.
7. 최종적으로 `final-summary.md`에 최선 후보, 주요 실패, 남은 후속 작업을 적는다.

## 실패 정책

- 실패가 명백히 기계적이고 후보 의도가 여전히 타당할 때만 재시도한다.
- 후보당 재시도는 보통 1-2회 이내로 제한한다.
- 타임아웃도 결과의 일부로 취급한다.
- 크래시와 타임아웃은 기록에서 숨기지 않는다.

## 패턴 조합

- `Pipeline`: 기준선 -> 변경 -> 평가 -> 판단
- `Supervisor`: 실험 후보 큐가 동적으로 바뀔 때
- `Producer-Reviewer`: 새 기준선 승격 전에 리뷰 승인이 필요할 때
