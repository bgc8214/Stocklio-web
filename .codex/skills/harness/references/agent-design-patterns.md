# Harness 아키텍처 패턴

Harness는 여섯 가지 재사용 가능한 아키텍처 패턴을 지원한다. 중요한 기준은 "가장 화려한 구조"가 아니라 "품질을 해치지 않으면서도 가장 작고 이해하기 쉬운 구조"다.

## 협업 방식 먼저 고르기

| 협업 방식 | 적합한 상황 | 대표 산출물 |
| --- | --- | --- |
| 단일 에이전트 | 전문 경계가 필요 없고 한 흐름으로 끝남 | `SKILL.md` 1개, 출력 계약 1개 |
| 순차 오케스트레이션 | 단계 의존성이 뚜렷함 | 오케스트레이터, `_workspace/` 핸드오프 |
| 병렬 팬아웃 | 독립적인 분기를 동시에 처리 가능 | 분기별 산출물, 통합 단계 |
| 감독자 중심 위임 | 작업 큐가 계속 바뀜 | 팀 스펙, 작업 큐, 재할당 규칙 |
| 계층적 위임 | 문제를 얕은 트리로 쪼개는 게 자연스러움 | 상위 오케스트레이터 + 하위 조정 레이어 |

판단 기준:

- 역할 경계가 재사용 가능하지 않으면 단일 에이전트에서 시작한다.
- 한 단계 산출물이 다음 단계 입력이 되면 순차 오케스트레이션으로 올린다.
- 분기끼리 진짜로 독립적일 때만 병렬 팬아웃을 쓴다.
- 처음부터 작업을 다 배치할 수 없으면 감독자 방식을 고려한다.
- 두 단계보다 깊은 계층은 대체로 평평하게 펴는 편이 낫다.

## 제거 가능한 하네스 규칙

- 핵심 워크플로우는 안정적인 산출물에 남기고, 모델별 재시도나 임시 복구 로직은 제거 가능한 섹션으로 분리한다.
- 특수 규칙이 많아질수록 구조를 더하는 대신 먼저 단순화할 기회를 찾는다.
- 숨겨진 런타임 마법보다 명시적인 파일 핸드오프가 낫다.

## 1. Pipeline

### 잘 맞는 상황

- 요구사항, 설계, 구현, 검증이 순서대로 이어짐
- 다음 단계가 이전 산출물 없이는 진행할 수 없음
- 중간 단계가 생략되면 품질이 크게 흔들림

### 맞지 않는 상황

- 여러 단계가 동시에 독립적으로 가능함
- 순차보다 나중 합성이 더 중요함
- 한 단계가 사실상 여러 unrelated 작업의 분배기 역할을 함

### 최소 산출물

- `.agents/skills/{domain}-orchestrator/SKILL.md` 또는 `docs/harness/{domain}/team-spec.md`
- `_workspace/01_*`, `_workspace/02_*` 같은 결정적 단계 산출물
- 반복 재사용되는 단계에만 전문 스킬

### 권장 방식

- 순차 오케스트레이터를 쓴다.
- 단계 경계마다 파일 핸드오프를 둔다.
- 다음 단계가 빠르게 읽을 수 있게 산출물을 좁게 유지한다.
- 실제 품질 게이트가 있을 때만 리뷰/QA 단계를 추가한다.

## 2. Fan-out/Fan-in

### 잘 맞는 상황

- 같은 입력을 여러 전문가가 서로 방해 없이 다룰 수 있음
- 다양한 관점 커버리지가 중요함
- 최종 결과가 여러 관점의 합성에서 이득을 봄

### 맞지 않는 상황

- 분기들이 서로의 중간 결과에 의존함
- 합성 단계가 문맥을 다시 다 복원해야 함
- 분기 소유권이 흐릿해서 결정적인 출력물을 정의하기 어려움

### 최소 산출물

- 분기 소유권이 있는 오케스트레이터 또는 팀 스펙
- 병렬 분기별 `_workspace/{phase}_{role}_{artifact}.md`
- 분기 결과를 인용하는 최종 통합 산출물

### 권장 방식

- 독립성이 명확한 분기만 병렬화한다.
- 모든 분기는 같은 입력 스냅샷을 기준으로 삼는다.
- 합성 기준을 작업 시작 전에 정의한다.
- 분기 산출물을 모두 보존해 나중에 의견 차이를 추적할 수 있게 한다.

## 3. Expert Pool

### 잘 맞는 상황

- 요청마다 필요한 전문가 조합이 달라짐
- 매번 모든 전문가를 돌리는 것보다 분류가 쉬움
- 이미 분리된 재사용 가능한 전문 스킬이 있음

### 맞지 않는 상황

- 거의 모든 요청이 거의 모든 전문가를 필요로 함
- 라우팅 기준이 너무 애매해서 역할이 계속 겹침
- 전문가가 지나치게 잘게 쪼개져 라우터 유지가 더 어려워짐

### 최소 산출물

- `docs/harness/{domain}/team-spec.md`의 라우팅 규칙
- `.agents/skills/` 아래 전문 스킬
- 필요하면 요청-스킬 매핑 표

### 권장 방식

- 라우팅 규칙을 명시적이고 결정적으로 쓴다.
- 어떤 요청에 어떤 전문가가 켜지는지 적는다.
- 애매한 경우를 위한 fallback 경로는 1개면 충분하다.
- 시간이 지나며 중복 스킬이 생기지 않는지 정기적으로 본다.

## 4. Producer-Reviewer

### 잘 맞는 상황

- 품질이 중요해서 별도 검토가 값어치를 함
- 리뷰 기준을 문서로 적을 수 있음
- 리뷰 결과를 승인, 수정, 재생성 정도로 제한할 수 있음

### 맞지 않는 상황

- 검토 기준이 너무 모호해 리뷰어가 두 번째 제작자가 됨
- 작업이 너무 작아 검토 루프가 과함
- 승인 기준이 없어 무한 수정 루프가 생김

### 최소 산출물

- 생산 스킬
- 리뷰 스킬 또는 QA 브리프
- `_workspace/` 리뷰 산출물
- 제한된 재시도 정책

### 권장 방식

- 리뷰 결과를 `pass`, `fix`, `redo`처럼 구조화한다.
- 재시도 횟수를 작게 제한한다.
- 리뷰어는 원래 요청과 산출물을 둘 다 읽어야 한다.
- 리뷰 메모는 일시적 채팅이 아니라 파일 산출물로 남긴다.

## 5. Supervisor

### 잘 맞는 상황

- 작업이 실행 중 계속 생기거나 바뀜
- 작업 단위 크기와 복잡도가 들쑥날쑥함
- 재할당과 우선순위 변경이 예외가 아니라 설계의 일부임

### 맞지 않는 상황

- 처음부터 작업을 깔끔하게 배치할 수 있음
- 감독자가 사실상 고정 체크리스트를 반복하는 수준임
- 관리할 백로그나 작업 인벤토리가 없음

### 최소 산출물

- 감독자 중심 팀 스펙
- 작업 큐 또는 작업 인벤토리
- 재할당/에스컬레이션 정책
- 최종 통합 또는 감사 보고서

### 권장 방식

- 감독 로직은 상위 오케스트레이터나 팀 스펙에 둔다.
- 작업 큐 형식은 마크다운이나 JSON으로 명시한다.
- 너무 잘게 쪼개지 말고 의미 있는 크기로 배치한다.
- 재할당이나 블로킹이 생기면 상태 업데이트를 명시적으로 남긴다.

## 6. Hierarchical Delegation

### 잘 맞는 상황

- 문제 자체가 도메인별 하위 흐름으로 자연스럽게 나뉨
- 단일 조정 레이어로는 너무 복잡함
- 하위 레이어가 상위에 안정적인 산출물을 돌려줄 수 있음

### 맞지 않는 상황

- 계층이 두 단계보다 깊어질 것 같음
- 하위 레이어가 사실상 스킬 하나로 충분함
- 같은 정보가 여러 층에 중복 복사됨

### 최소 산출물

- 상위 오케스트레이터 또는 팀 스펙
- 하위 조정 메모 또는 역할 브리프
- 부모-자식 간 핸드오프 산출물

### 권장 방식

- 계층은 얕게 유지한다.
- 최상위는 전체 목표와 승인 기준을 가진다.
- 하위 레이어는 각자 뚜렷한 하위 도메인을 맡는다.
- 보고 체계가 이해를 돕기보다 숨기기 시작하면 평평하게 바꾼다.

## 패턴 조합

- `Pipeline + Fan-out/Fan-in`: 순차 단계 중 일부만 병렬 분기
- `Fan-out/Fan-in + Producer-Reviewer`: 병렬 생성 후 리뷰/합성
- `Supervisor + Expert Pool`: 감독자가 상황에 맞는 전문가를 선택
- `Hierarchical Delegation + Pipeline`: 상위 분해 후 각 하위 도메인 내부는 순차 흐름

## 워크플로우 프로필: 자율 실험

자율 실험은 일곱 번째 아키텍처 패턴이 아니다. 좁은 변경 표면을 반복 실험하고, 고정된 평가 표면과 지표로 비교하는 워크플로우 프로필이다.

핵심 규칙:

- 첫 실험 전에 변경 가능한 표면을 선언한다.
- 실행 중 평가 표면은 읽기 전용으로 유지한다.
- 변이 전에 측정된 기준선을 만든다.
- 크래시와 타임아웃도 기록한다.
- 기본은 사용자가 통제하는 컴퓨트 위에서 돌린다.

## 산출물 형태 규칙

- 재사용 가능한 조정 로직은 `.agents/skills/{domain}-orchestrator/SKILL.md`
- 역할 구조와 핸드오프는 `docs/harness/{domain}/team-spec.md`
- 재사용 가능한 역할 행동은 `.agents/skills/{specialist}/SKILL.md`
- 긴 도메인 설명은 `.agents/skills/{specialist}/references/`
- 모델별 복구 메모는 제거 가능한 참고 문서
- 중간 산출물은 `_workspace/`에 결정적인 이름으로 저장
- 자율 실험 장부는 `_workspace/experiments/{run}/`
