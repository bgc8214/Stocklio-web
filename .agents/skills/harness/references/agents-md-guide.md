# 좋은 `AGENTS.md` 쓰기

`AGENTS.md`는 거의 모든 세션에서 읽히는 저장소 안내문이다. 코드베이스에서 아는 것을 전부 쏟아붓는 곳이 아니라, 반복적으로 도움이 되는 핵심 규칙만 남기는 곳으로 다뤄야 한다.

## 핵심 원칙

짧게, 사람이 직접 쓴 톤으로, 저장소 전반에 걸쳐 거의 모든 작업에서 도움이 되는 내용만 넣는다.

특정 워크플로우에만 필요한 내용이라면 `.agents/skills/`, 팀 스펙, 상세 문서로 옮기고 `AGENTS.md`에서는 링크만 남긴다.

## 무엇을 넣을까

- `WHAT`: 프로젝트 목적, 스택 선택, 기준 경로, 혼동하면 안 되는 핵심 표면
- `WHY`: 프로젝트가 존재하는 이유, 중요한 경계가 왜 중요한지
- `HOW`: 정확한 빌드/테스트/검증 명령과 `uv`, `bun`처럼 비직관적인 도구 선택 이유
- 조건부이거나 역할별인 깊은 문서로 가는 포인터

## 무엇을 빼야 할까

- 자동 생성된 디렉터리 덤프
- 다른 에이전트가 스스로 찾을 수 있는 코드베이스 관광 가이드
- 포매터, 린터, 테스트가 이미 강제하는 스타일 규칙
- 가끔만 필요한 작업별 플레이북
- 잠깐 쓰는 모델별 재시도/복구 로직

## Progressive Disclosure

- `AGENTS.md`는 안정적인 입구로 둔다.
- 자세한 워크플로우는 `.agents/skills/`, `docs/`, 저장소 로컬 참조 문서로 보낸다.
- 복붙보다 링크를 선호해 유지비를 줄인다.

## 제거 가능한 하네스 메모

`AGENTS.md`는 저장소 자체의 안정적인 규칙을 적는 곳이다. 모델별 편법, 프롬프트 보정, 임시 휴리스틱은 오케스트레이터 스펙이나 연결된 참고 문서에 두고, 필요 없어지면 쉽게 지울 수 있어야 한다.

## Compact Template

```markdown
# Repository Agents Guide

Keep this file short and repo-wide. Point to deeper docs when detail is conditional.

## WHAT
- {프로젝트 목적과 핵심 표면}
- {기준 경로 또는 공유 계약}
- {비직관적인 도구나 런타임 선택}

## WHY
- {프로젝트가 존재하는 이유}
- {주요 경계나 제약이 중요한 이유}

## HOW
- {정확한 빌드, 테스트, 검증 명령}
- {문서 동기화나 배포 관련 규칙}
- {깊은 문서로 가는 포인터}
```

## 빠른 자가 점검

- 이 문장이 대부분의 작업에 도움이 되는가
- 저장소를 이해한 사람이 유지할 의도로 쓴 문장인가
- 세부사항은 연결 문서로 빼는 편이 낫지 않은가
- 모델 성능이 좋아져도 이 규칙은 여전히 `AGENTS.md`에 남아야 하는가
