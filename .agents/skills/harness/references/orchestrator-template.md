# 오케스트레이터 스펙 템플릿

워크플로우가 충분히 재사용 가능해 상위 오케스트레이터 스킬이나 팀 스펙이 필요할 때 이 템플릿을 쓴다.

## 작성 원칙

- 역할보다 목표를 먼저 적는다.
- 각 단계의 출력물과 핸드오프 파일 이름을 모두 명시한다.
- 암묵적 조정보다 마크다운 스펙과 `_workspace/` 산출물을 선호한다.
- 병렬화는 범위가 분명한 경우에만 적는다.
- 실패 시 어떻게 할지 명시한다.
- 모델별 재시도나 임시 편법은 지워도 되는 섹션으로 분리한다.

## 템플릿

```markdown
# {도메인} 오케스트레이터

## 목표
이 오케스트레이터가 책임지는 끝단 결과와 범위를 적는다.

## 입력
- 필요한 저장소, 문서, 사용자 입력
- 정보가 없을 때 허용하는 가정

## 출력
- 최종 산출물
- 반드시 보존해야 하는 중간 산출물

## 역할
| 역할 | 책임 | 재사용 스킬 | 작성 파일 |
| --- | --- | --- | --- |
| {role-1} | {무엇을 책임지는가} | `.agents/skills/{role-1}/SKILL.md` 또는 `n/a` | `_workspace/01_{role-1}_{artifact}.md` |
| {role-2} | {무엇을 책임지는가} | `.agents/skills/{role-2}/SKILL.md` 또는 `n/a` | `_workspace/02_{role-2}_{artifact}.md` |

## 단계 순서

### Phase 1: {이름}
- 입력 소스:
- 작업:
- 출력 파일:
- 완료 기준:

### Phase 2: {이름}
- 입력 소스:
- 작업:
- 출력 파일:
- 완료 기준:

### Phase 3: {이름}
- 입력 소스:
- 작업:
- 출력 파일:
- 완료 기준:

## 핸드오프 파일
| 보낸 역할 | 받는 역할 | 파일 | 목적 |
| --- | --- | --- | --- |
| {role-1} | {role-2} | `_workspace/01_{role-1}_{artifact}.md` | {왜 이 핸드오프가 필요한가} |

## 실패 정책
- 재시도 정책:
- 부분 완료 정책:
- 충돌 해결 정책:
- 에스컬레이션 조건:

## 제거 가능한 모델 전용 로직
- 임시 재시도/복구 휴리스틱:
- 선택적 지름길 또는 가드레일:
- 삭제 기준:

## 검증 체크
- 구조 체크:
- 내용 체크:
- 시나리오 테스트:

- 안전한 병렬 분기:
- 겹치면 안 되는 영역:
- 최종 통합 책임자:

## 테스트 시나리오
### 정상 흐름
- 요청:
- 기대 단계 산출물:

- 기대 fallback 동작:
```

## 권장 `_workspace/` 구조

```text
_workspace/
├── 00_input/
│   └── request-summary.md
├── 01_{role}_{artifact}.md
├── 02_{role}_{artifact}.md
├── 03_review_{artifact}.md
└── final/
    └── {deliverable}.md
```
