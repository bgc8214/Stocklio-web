# 하네스

Harness는 저장소 안에서 재사용 가능한 작업 체계를 설계하는 메타 스킬이다. 프로젝트를 분석하고, 어떤 안내가 `AGENTS.md`에 있어야 하는지 판단하고, 협업 패턴을 고르고, 전문 스킬과 팀 스펙을 만들고, 그 사이의 핸드오프 파일 구조까지 정한다.

## 언제 사용할까

- 특정 도메인용 스킬 묶음을 새로 설계할 때
- 기존 작업 흐름을 저장소 로컬 공유 스킬로 바꿀 때
- 역할 분리, 오케스트레이션 규칙, 검증 단계를 재사용 가능한 형태로 만들고 싶을 때
- 복잡한 작업을 위해 파일 이름, 핸드오프 문서, 리뷰 루프를 표준화하고 싶을 때

기존 스킬 하나로 충분한 단발성 작업에는 사용하지 않는다.

## 필요한 입력

- 도메인 또는 프로젝트 목표
- 주요 작업 흐름과 최종 산출물
- 제약 사항, 품질 기준, 실패 허용 범위
- 이미 존재하는 코드베이스나 문서 맥락
- 재사용할 만한 기존 스킬, 역할 문서, 템플릿

정보가 비어 있으면 저장소를 먼저 살펴보고, 저장소 로컬 문맥에 맞는 가장 좁고 안전한 가정을 둔다.

## 생성 산출물

- 저장소 전체 규칙이 필요할 때만 `AGENTS.md`
- 최상위 오케스트레이션용 `.agents/skills/{domain}-orchestrator/SKILL.md`
- 전문 역할용 `.agents/skills/{specialist}/SKILL.md`
- 깊은 참고 정보용 `.agents/skills/{specialist}/references/*`
- 역할 구조, 핸드오프, 실패 정책을 담는 `docs/harness/{domain}/team-spec.md`
- 전체 스킬까지는 필요 없지만 역할 문서가 필요한 경우 `docs/harness/{domain}/roles/{role}.md`
- 중간 산출물과 리뷰 근거를 남기는 `_workspace/{phase}_{role}_{artifact}.md`
- 자율 실험 루프가 있을 때 `_workspace/experiments/{run}/results.tsv`

기본값은 "전문 스킬 + 팀 스펙"이다. 역할이 반복적으로 재사용될 때만 역할 전용 파일을 더 만든다.

## AGENTS.md 작성 원칙

`AGENTS.md`는 거의 모든 세션에서 읽히는, 가장 영향력이 큰 저장소 안내 문서다. 그래서 정말 저장소 전체에 걸쳐 반복적으로 필요한 내용만 담아야 한다.

- 짧고 사람이 쓴 문체를 유지한다.
- `WHAT / WHY / HOW` 같은 저장소 전반 규칙에만 집중한다.
- 프로젝트 목적, 기준 경로, 혼동하면 안 되는 경계, 정확한 빌드/테스트/검증 명령을 넣는다.
- 조건부이거나 특정 역할에만 필요한 세부사항은 깊은 문서로 빼고 링크만 남긴다.
- 코드베이스 관광 가이드나 자동 생성된 디렉터리 덤프는 넣지 않는다.
- `AGENTS.md`를 쓸 때는 `references/agents-md-guide.md`를 먼저 읽는다.

## 하네스 설계 원칙

- 프롬프트 문구만으로 모든 것을 해결하려 하지 말고, 역할 계약과 산출물 계약을 분명히 만든다.
- 모델별 재시도, 임시 휴리스틱, 복구 로직은 지워도 되는 섹션이나 참조 문서로 분리한다.
- 협업은 얕게 유지한다. 지나치게 깊은 라우팅이나 복잡한 복구 규칙이 필요하면 설계를 먼저 단순화한다.

## 기본 권장값

- 저장소 로컬 스킬을 우선한다.
- 기본은 단일 메인 에이전트다.
- 워커는 범위가 뚜렷하고 병렬화 이득이 분명할 때만 분리한다.
- 런타임 내부 메시징 대신 파일 핸드오프와 마크다운 스펙을 선호한다.
- 저장소가 원래 의존하지 않는다면 모델 고정, SDK 런타임, MCP 오케스트레이션을 전제로 삼지 않는다.
- 생성되는 모든 `SKILL.md`는 YAML frontmatter로 시작해야 한다.
- frontmatter에는 최소 `name`과 `description`이 있어야 한다.
- 스킬 이름은 안정적이고 저장소 친화적으로 유지한다.

## 6단계 워크플로우

### Phase 1: 도메인 분석

1. 저장소, 요청, 기존 문서를 살핀다.
2. 도메인, 핵심 작업 유형, 기대 산출물, 품질 기준을 정리한다.
3. 이미 있는 재사용 가능한 문서, 스킬, 경로 계약을 기록한다.
4. 이 작업을 스킬로 만들지, 역할 문서로 만들지, 오케스트레이터로 만들지 판단한다.
5. 자율 실험 루프 요청이라면 변경 가능한 표면, 고정 평가 표면, 기준선, 측정 지표를 먼저 정한다.
6. 새 산출물을 만들기 전에 도메인 요약을 남긴다.

산출물:

- 도메인 요약
- 작업 인벤토리
- 재사용 가능한 기존 자산 메모

### Phase 2: 팀 아키텍처 설계

1. 가장 작은 구조로 문제를 덮을 수 있는지 판단한다.
2. 단일 에이전트, 순차 오케스트레이션, 제한적 병렬 분기 중 무엇이 맞는지 정한다.
3. `references/agent-design-patterns.md`에서 여섯 가지 패턴 중 하나를 고른다.
4. 자율 실험 루프라면 `references/autonomous-experimentation.md`의 프로필과 조합 방식을 정한다.
5. 단계 사이를 어떻게 `_workspace/` 파일로 넘길지 정한다.
6. 제거 가능한 복구 로직과 핵심 계약을 분리한다.

산출물:

- 선택한 패턴
- 역할 목록
- 핸드오프 계획
- 산출물 이름 규칙

### Phase 3: 역할과 산출물 정의 생성

안정적인 역할은 다음 중 하나로 만든다.

- 재사용 가능한 전문 스킬
- 재사용 가능한 오케스트레이터 스킬
- `docs/harness/{domain}/roles/` 아래 역할 문서

각 역할에는 책임, 입력, 출력, 검토 경계, 실패 시 행동을 명시한다. 너무 좁거나 일회성인 역할은 파일로 분리하지 않는다.

산출물:

- 역할 인벤토리
- 파일 배치 계획
- 역할별 입력/출력 계약

### Phase 4: 스킬 생성

1. 재사용 가능한 스킬을 `.agents/skills/` 아래 만든다.
2. 모든 `SKILL.md`는 YAML frontmatter 뒤에 본문이 와야 한다.
3. 본문은 가볍게 유지하고, 긴 설명은 `references/`로 뺀다.
4. `When to Use`, `Required Inputs`, `Workflow`, `Outputs`, `Validation`을 가능한 한 포함한다.
5. 반복되는 검증이나 구조 체크를 줄여줄 때만 보조 스크립트를 번들링한다.

산출물:

- 전문 스킬
- 필요하면 오케스트레이터 스킬
- 단계적 로딩용 참조 문서

### Phase 5: 통합과 오케스트레이션

1. 전체 흐름을 오케스트레이터 스킬이나 팀 스펙으로 정의한다.
2. 단계 순서, 핸드오프 파일, 소유권, 실패 시 대처, 제거 가능한 휴리스틱을 적는다.
3. 병렬화는 진짜로 독립적인 분기에서만 사용한다.
4. 자율 실험 루프라면 `_workspace/experiments/{run}/` 아래 결과 장부와 기준선을 유지한다.
5. 중간 산출물은 디버깅과 감사 추적을 위해 보존한다.

산출물:

- 오케스트레이터 스킬 또는 팀 스펙
- `_workspace/` 계약
- 실패 및 재시도 정책

### Phase 6: 검증과 테스트

1. 경로와 내부 링크를 검증한다.
2. 정상 흐름과 실패 흐름을 각각 최소 1개 이상 테스트한다.
3. 자율 실험 루프라면 기준선, 고정 평가 표면, 결과 장부, 타임아웃/크래시 기록까지 확인한다.
4. 필요하면 스킬 사용 버전과 비사용 버전을 비교한다.
5. 모호함, 과도한 복잡성, 쓸모없는 지시가 보이면 단순화한다.

산출물:

- 검증 체크리스트
- 테스트 시나리오
- 수정 또는 단순화 메모

## 아키텍처 선택

품질과 이해 가능성을 해치지 않는 가장 작은 패턴을 고른다.

| 패턴 | 잘 맞는 상황 | 기본 구현 방식 |
| --- | --- | --- |
| Pipeline | 순차 의존 작업 | 오케스트레이터 + `_workspace/` 핸드오프 |
| Fan-out/Fan-in | 병렬 조사 후 합성 | 병렬 분기 + 최종 통합 |
| Expert Pool | 요청별로 다른 전문가 선택 | 팀 스펙의 라우팅 규칙 + 전문 스킬 |
| Producer-Reviewer | 생성 후 명시적 품질 검토 | 생산 스킬 + 리뷰 스킬 + 제한된 수정 루프 |
| Supervisor | 동적으로 바뀌는 백로그 관리 | 상위 조정자 + 재할당 규칙 |
| Hierarchical Delegation | 자연스러운 계층 분해 | 얕은 계층 구조 + 하위 조정 레이어 1단 |

## 워크플로우 프로필

### 자율 실험

- 일곱 번째 아키텍처 패턴이 아니다.
- 사용자가 직접 통제하는 컴퓨트 위에서 좁은 변경 표면을 반복 실험할 때 사용한다.
- 단순한 기준선 -> 변경 -> 평가 -> 채택/폐기 루프는 `Pipeline`과 잘 맞는다.
- 실험 후보 큐가 계속 바뀌면 `Supervisor`와 잘 맞는다.
- 자세한 규칙은 `references/autonomous-experimentation.md`를 본다.

## 검증 기대값

- 경로와 내부 참조가 실제와 맞는다.
- 전문 스킬과 팀 스펙이 같은 산출물 이름을 사용한다.
- 각 단계마다 이름 있는 출력물이 있다.
- 품질 리스크가 높으면 리뷰 또는 QA 단계가 명시된다.
- `When to Use`와 `Required Inputs`가 겹치지 않게 구체적이다.
- 모든 생성 스킬은 `name`, `description`이 있는 YAML frontmatter로 시작한다.
- `_workspace/` 핸드오프는 예측 가능하고 보존된다.
- 저장소 `AGENTS.md`는 짧고 링크 중심이다.
- 모델별 복구 로직은 전체 구조를 다시 쓰지 않고도 제거할 수 있다.
